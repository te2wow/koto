kotovstakt

どちらも「AIにプロセスを守らせる」ためのYAMLワークフローツール。
同じ哲学から、koto は別の割り切りを選んだ。

指揮棒(takt) と 和楽器(koto) — 拍を刻むという同じ語源から

一行でいうと

🎼 takt

nrslib/takt — TypeScript / npm

  • 複数のAIエージェントをYAMLワークフローで強制制御するオーケストレーター
  • Persona / Policy / Knowledge / Instruction / Output Contract に分けた Faceted Prompting
  • 並列レビュー・多数決(MAGI)・複数プロバイダSDK対応の多機能フレームワーク
  • レビューは「AIレビュアーの判断」で承認/差し戻し

🎴 koto

te2wow/koto — Go single binary

  • CLIコーディングエージェントをYAMLワークフローで駆動するミニマルなランナー
  • 1ステップ = 1プロンプト。読めば全部わかる
  • 品質ゲートを実コマンドの終了コードで強制。緑になるまで止まらない
  • レビューは「go test が通るか」という偽れない外部信号で判定
takt へのリスペクト。 「AIを信頼しつつ、プロセスは仕組みで担保する」という発想は takt が切り拓いたものです。koto はその哲学をそのまま受け継ぎ、配布形態・強制の手段・機能の絞り方 という3点だけを別の方向に振り切った姉妹的な実装です。対立ではなく、設計上の選択の違いです。

核心の違い

何が「次に進んでよい」を決めるのか

takt は AIレビュアーの判断でループを回す。koto は 実際に走らせたコマンドの終了コードでループを回す。テストは通るか通らないかで、 モデルがごまかせない。これが最大の差別化点。

観点taktkoto
配布形態 TypeScript / npm
Node とdependency tree が必要		 Go 単一バイナリ・依存ゼロ
go install 一発 / バイナリDL
ループを強制する仕組み AIレビュアーの「approved / needs fix」判断 実コマンドの exit code（go test, lint, pytest）
プロンプト構成 Faceted Prompting
（persona/policy/knowledge/instruction/output を5ファイルに分割）		 1ステップ = 1プロンプト
（YAML内にそのまま書く）
対応プロバイダ SDK + CLI（6種） CLI exec のみ
SDK更新に振り回されない
合議・並列 MAGI多数決・並列レビュー集約 直列ループ中心（必要十分に絞る）
信頼性の土台 レビューループ レビューループ + 実行ゲート + 外部フィードバック注入
コード規模 大きめのTSフレームワーク 小さく読めるGo（半日で全部追える）
哲学 共通: AIを信頼しつつ、プロセスは仕組みで担保する

なぜ「実行ゲート」なのか

学術的にも裏付けがある: 外部フィードバックのない自己反省は、むしろ精度を下げうる (ICLR'24)。同一モデルの自問自答ループは「思考の退化」を招く。だからレビューの判定は、 テスト・lint・型チェックといった外部の検証信号に錨を下ろすべき—— これが koto がゲートを「AIの判断」ではなく「コマンドの終了コード」に置いた理由。

koto の実行ログ（実機で claude を駆動した記録）:

▶ implement (agent)   → gate
▶ gate (gate)         ✗ gate failed (exit 1, attempt 1/6)   → fix
▶ fix (agent)        → gate
▶ gate (gate)         ✓ gate passed
✓ complete

YAMLの書き味

🎼 takt

name: code-review
personas:
  coder: ../personas/coder.md
  supervisor: ../personas/supervisor.md
steps:
  - name: write-code
    persona: coder
    rules:
      - condition: 実装完了
        next: review

ペルソナは別ファイル。関心の分離が効く反面、全体像は複数ファイルを横断して読む。

🎴 koto

- name: test
  type: gate
  run: "{{vars.test_cmd}}"   # 実コマンド
  max_retries: 5
  on_pass: COMPLETE          # exit 0
  on_fail: fix               # 非0→修正へ

ゲートは1ステップで完結。「何が緑判定か」が run に直接書いてある。

どちらを選ぶか

takt が向く人

  • 複数エージェントの並列・合議を本格的に組みたい
  • ペルソナ・ポリシーを体系的に再利用したい
  • Node/TS環境が既にあり、多機能さを活かせる
  • レビューの質的判断そのものをAIに任せたい

koto が向く人

  • 依存ゼロの単一バイナリでサッと入れたい（CI含む）
  • 「テストが緑になるまで直し続ける」が主目的
  • ワークフローを1ファイルで把握したい
  • 判定を偽れない客観信号（exit code）に委ねたい
補足: koto は takt の劣化コピーではなく、「軽量・実行ゲート・CLI exec」という別の最適化点を選んだ実装です。 多機能・合議・体系的プロンプト管理が要るなら takt、単一バイナリで品質ゲートを機械的に強制したいなら koto—— という棲み分けになります。
